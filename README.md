# Проект: ETL-пайплайн на Apache Spark
## Описание
Проект демонстрирует создание ETL-пайплайна для обработки данных о кликах пользователей с использованием Apache Spark.
Генерируются тестовые данные с «плохими» записями, затем выполняются очистка, трансформация и агрегация. Результаты сохраняются в структурированном виде для дальнейшего анализа.

## Структура проекта
- data/ – исходные данные (clickstream.csv), создаваемые генератором.
- src/ – исходный код:
    - generate_data.py – генератор тестовых данных (55 000 записей, 10% с ошибками).
    - etl_pipeline.py – основной ETL-пайплайн (очистка, трансформация, агрегация, сохранение).
    - verify_results.py – скрипт для проверки целостности результатов.
- output/ – результаты выполнения (очищенные данные, агрегаты, отчёты). Каждый запуск создаёт новую папку с временной меткой.
- docs/ – документация и скриншоты (для отчёта).
- requirements.txt – зависимости проекта.

## Требования
- Python 3.11 (или 3.13 с совместимыми версиями пакетов)
- JDK 11
- Apache Spark 3.5.8

## Установка и запуск
- Клонируйте репозиторий:

```powershell
git clone https://github.com/your-username/spark-etl-project.git
cd spark-etl-project
```
- Создайте виртуальное окружение и установите зависимости:

```powershell
python -m venv venv
.\venv\Scripts\activate   # для Windows
pip install -r requirements.txt
```
- Запустите Spark-кластер (локальный режим standalone):

    - Откройте два окна PowerShell.
    - В первом выполните:

    ```powershell
    cd C:\spark\spark-3.5.8-bin-hadoop3
    .\bin\spark-class.cmd org.apache.spark.deploy.master.Master
    ```

    - Во втором (после запуска мастера):

    ```powershell
    cd C:\spark\spark-3.5.8-bin-hadoop3
    .\bin\spark-class.cmd org.apache.spark.deploy.worker.Worker spark://localhost:7077
    ```
- Сгенерируйте тестовые данные:

```powershell
cd src
python generate_data.py
```
- Запустите ETL-пайплайн:

```powershell
python etl_pipeline.py
```
Проверьте результаты в папке output/.

## Как это работает (подробно)
### 1. Генерация данных (generate_data.py)

Использует библиотеки pandas, numpy и faker для создания 55 000 записей.

90% записей – «хорошие»: корректные идентификаторы, даты, цены и т.д.

10% – «плохие»: пустые user_id, отрицательные duration_sec, неверные форматы дат, неизвестные устройства и т.п. Это нужно для проверки устойчивости пайплайна.

Данные сохраняются в data/clickstream.csv с кодировкой UTF-8.

### 2. ETL-пайплайн (etl_pipeline.py)

Класс SparkETLPipeline реализует стандартные этапы ETL.

Инициализация SparkSession
Подключается к мастеру (spark://192.168.10.58:7077).

### 3. Этап EXTRACT (extract())

Загружает CSV с заданной схемой.

Опция PERMISSIVE и columnNameOfCorruptRecord позволяет Spark помечать строки, которые не удалось распарсить (например, если текстовое поле попало в числовую колонку). Такие строки попадают в виртуальную колонку _corrupt_record.

Если _corrupt_record отсутствует, значит, все строки формально соответствуют схеме. Однако это не означает, что данные идеальны – просто они не ломают парсинг. Проблемы вроде пустых строк или отрицательных чисел будут обработаны позже.

### 4. Этап TRANSFORM (transform())

- Очистка:

Удаление дубликатов по user_id, session_id, timestamp.

Замена null на значения по умолчанию (например, region = "Неизвестно").

Фильтрация записей с пустым user_id, отрицательной длительностью или ценой. 

- Обогащение:

Из строки timestamp создаются колонки event_timestamp (тип timestamp), event_date, event_hour, event_dayofweek.

Категоризация сессий по длительности (short, medium, long) и цен (low, medium, high).

После очистки остаётся около 53 144 записей (из 55 000). Отбрасывается примерно 1 856 «грязных» строк.

### 5. Этап ANALYZE (analyze())

Выполняет четыре агрегации:

activity_by_region_hour – активность по регионам и часам (кол-во событий, уникальные пользователи, средняя длительность, выручка).

device_statistics – статистика по типам устройств и категориям сессий.

top_users – топ-100 пользователей по числу сессий (без оконной функции, чтобы избежать предупреждений).

daily_activity – ежедневная активность (события, пользователи, выручка).

### 6. Этап LOAD (load())

Создаётся уникальная папка output/run_<timestamp>.

Очищенные данные сохраняются в формате Parquet с партиционированием по event_date. Parquet выбран за оптимальную производительность и сжатие.

Агрегированные данные сохраняются дважды:

В Parquet (для возможного дальнейшего анализа).

В CSV (один файл на отчёт) для удобного просмотра.

Текстовый отчёт execution_report.txt содержит статистику выполнения и примеры данных.

После каждой записи вручную удаляются временные папки _temporary, которые Spark создаёт в процессе. Это гарантирует, что на диске не остаётся мусора.

### 7. Проверка результатов (verify_results.py)

Запускает локальную Spark-сессию и проверяет наличие всех ожидаемых файлов.
Выводит количество записей в каждом наборе данных и примеры строк.
Убеждается, что пайплайн отработал корректно.

## Почему в консоли появляются красные строки (WARN)?
Во время выполнения вы можете увидеть множество сообщений типа:

```text
WARN FileUtil: Failed to delete file or dir ... it still exists.
WARN TransportChannelHandler: Exception in connection ...
```
Это предупреждения (WARN), а не ошибки (ERROR). Они не влияют на корректность данных и возникают из-за особенностей работы Spark на Windows:

WARN FileUtil – Spark пытается удалить временные папки _temporary после записи, но мы уже удалили их вручную. Spark не находит файлы для удаления и выдаёт предупреждение. Фактически мусора на диске нет – мы это гарантируем.

WARN TransportChannelHandler – возникает при завершении сессии, когда соединения между драйвером и исполнителями обрываются. Это стандартное поведение на Windows и не влияет на уже записанные данные.

Такие предупреждения можно считать «ложными тревогами». Если они мешают, можно установить уровень логирования Spark на ERROR (раскомментировать строку в __init__), но тогда вы потеряете и другие предупреждения, которые могут быть полезны при отладке.

## Результаты
После выполнения в папке output/ появится директория с меткой времени, содержащая:

cleaned_data/clickstream_cleaned – очищенные данные в Parquet (с разбивкой по датам).

aggregated/ – агрегированные данные в Parquet.

reports/ – CSV-отчёты (по одному файлу на каждую агрегацию).

execution_report.txt – подробный отчёт о выполнении.

## Заключение
Проект демонстрирует полноценный ETL-цикл на Spark: генерация «грязных» данных, их очистка, обогащение, агрегация и сохранение. Все шаги автоматизированы, результаты проверяемы. Предупреждения в консоли не должны вызывать беспокойства – они связаны с окружением Windows и не влияют на качество обработки.
